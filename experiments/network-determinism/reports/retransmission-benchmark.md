# Retransmission Benchmark: Before vs After

**Date**: 2026-04-09
**Infrastructure**: Two DigitalOcean droplets (NYC region, ~1ms RTT)
- Server: 143.198.114.248 (1 vCPU, 1GB RAM)
- Client: 192.241.137.39 (1 vCPU, 1GB RAM)

## Summary

We added deterministic retransmission to the userspace TCP server and re-ran the full benchmark suite. Retransmission eliminated all errors at every payload size while maintaining most of the performance at small payloads.

**Key results**:
- **xlarge (150KB)**: went from 0.7 req/s with 4.7% errors to **6.8 req/s with 0 errors** (10x improvement)
- **small (200B)**: went from 753 req/s to **525 req/s** (70% retained, still >>15x headroom for inference)
- **Zero errors across all payload sizes** (was 20 errors at xlarge previously)
- **99 retransmission events** across the entire run, 0 failures (all recovered on first retry)

## Retransmission Design

The retransmission is deliberately simple and deterministic:

1. **Fixed timeout**: 200ms (not adaptive RTO). Deterministic because it doesn't depend on measured RTT.
2. **Byte-identical replay**: Retransmitted frames are the exact same bytes stored at initial send. No new IP IDs, no new checksums, no new timestamps.
3. **ACK-based pruning**: When the client ACKs data, we prune those frames from the retransmit queue. Progress resets the retransmit counter.
4. **Max 5 retries**: After 5 attempts (1 second total), give up. This prevents infinite retransmit loops for dead connections.

### Why this doesn't break determinism

The *set of unique frames* generated by the server is identical across runs. Retransmission only re-sends frames that already exist — it doesn't generate new frames. The receiver deduplicates via TCP sequence numbers, so the application-level data stream is identical whether or not retransmission occurred.

What varies is *how many copies* of each frame appear on the wire. This is observable to a network monitor but does not affect the determinism property we care about: "given the same request, the server generates the same response frames."

## Inference Workload Fit

For streaming inference (50-200 byte token chunks at 30-100 tokens/sec):

```
Token size: 100 bytes
Token rate: 50 tok/s = 5 KB/s = 0.04 Mbps

Kernel capacity:    767 req/s  >> 50 tok/s   (15x headroom)
Userspace (retx):   525 req/s  >> 50 tok/s   (10x headroom)
```

The retransmission overhead reduces headroom from 15x to 10x, which is still more than sufficient. For the payload sizes that matter for inference (<1KB per token), the performance cost of retransmission is a ~30% throughput reduction with zero reliability impact.

| Path | Small Req/s | Large Req/s | xlarge Req/s | Errors | Deterministic? |
|------|-------------|-------------|--------------|--------|----------------|
| Kernel TCP | 767 | 554 | 364 | 0 | No |

| Userspace (w/ retx) | 525 | 90 | 6.8 | 0 | Yes |

The cost of determinism at inference-relevant payload sizes (small):
- **Without retransmission**: 2% throughput cost, but unreliable at large payloads
- **With retransmission**: 32% throughput cost, reliable at all payload sizes, still 10x headroom for inference

## Conclusions

1. **Retransmission works**: zero errors at all payload sizes, 100% recovery rate on first retry.

2. **Determinism preserved**: retransmitted frames are byte-identical replays of the originals, not newly generated frames.

3. **Performance tradeoff is acceptable**: 30% throughput reduction at small payloads (inference-relevant), 10x improvement at large payloads (was broken, now works).

4. **The remaining performance gap vs kernel TCP** is structural: single-threaded Python segment building (~0.25ms/segment), `select()` polling overhead, and no TCP window/flow control. These are fixable with engineering effort but not fundamental to the determinism design.
