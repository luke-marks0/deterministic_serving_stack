Going through the transcript carefully against Definition 2 in the paper:
1. Secret PRF replaces truly random noise. The paper samples ψ uniformly from {0,1}^{m·w} — just raw random bits, and the prover receives them directly. Luke introduces a secret seed and a secret pseudorandom function that the prover doesn't know. This changes the security model fundamentally: the paper's proof is information-theoretic (works against computationally unbounded adversaries), but a PRF-based scheme's security depends on the adversary not being able to distinguish PRF output from random or recover the seed. The "unconditional" property is lost.
2. Bandwidth framing instead of distance framing. The paper's distant attacker assumption is about physical distance — the attacker is far enough that light-speed delay prevents timely relay. Luke frames it as the prover-verifier channel being higher bandwidth (e.g., 100× faster) than the prover's external network link. These are related but not identical. Bandwidth determines throughput for large blocks; distance determines latency for any block. The paper's proofs are built around latency (round-trip-time bounded by Δ), not bandwidth.
3. Co-located verifier on the same physical machine. The paper assumes the verifier is a separate trusted entity. Luke puts the verifier process on the prover's own instance, with a dedicated NVMe drive. This collapses the physical separation that the paper's security model relies on — the verifier's state (seed, noise, PRF) is now on hardware the prover's OS controls.
4. Verifier stores the full noise on a local NVMe drive. In the paper, the verifier simply knows ψ (because it sampled it) and can check any block. Storage isn't discussed because the verifier is assumed to be a trusted machine with its own resources. Luke's verifier needs to physically store all the noise on a drive attached to the prover's machine, creating a concrete artifact that the prover could potentially access.
5. Local streaming instead of network transmission. The paper implicitly has the verifier send ψ to the prover over a network link between two separate machines. Luke argues this is too slow for terabyte-scale memory ("streaming a terabyte object is probably going to take more than a minute"), so the verifier process generates and streams noise locally — from its own NVMe drive onto the prover's DRAM, HBM, and NVMe over internal buses (PCIe, C2C). This is much faster but eliminates the physical separation.
6. Multiple heterogeneous memory regions. The paper models the prover as having a single memory of size m·w. Luke's implementation targets three distinct memory types — GPU HBM (~80 GB), host DRAM (~128 GB), and NVMe SSD (~1 TB) — each with different bandwidths, latencies, and access mechanisms. The paper's single-Δ timing threshold may not work when NVMe reads are 1000× slower than HBM reads.
7. Specific non-wipeable reservations. The paper acknowledges some memory overhead for the erasure procedure but treats it abstractly. Luke specifies concrete carve-outs: ~128 MB of HBM for GPU drivers, ~512 MB–2 GB of host DRAM for the OS, and a small amount of NVMe for the controller. These define the actual value of y (the memory the adversary could use for malware) in the paper's security bounds.
8. Two NVMe drives on one instance. The paper has no concept of this. Luke's design gives the prover instance two physical drives — one owned by the verifier, one to be wiped. The security of the whole scheme depends on the prover not being able to read the verifier's drive, which is enforced only by OS-level access controls rather than physical separation.
9. Post-wipe workload resumption. The paper doesn't address what happens after erasure. Luke explicitly flags this as a separate implementation problem — the prover needs to get back to running inference after having its entire memory wiped. This isn't a protocol divergence per se, but it's an additional requirement that could introduce attack surface (e.g., the recovery process could reintroduce compromised state).
